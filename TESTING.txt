1. TOOL SELECTION
--------------------------------------------------------------------------------

Vitest is the test runner (not Jest) because:
  - It reuses the existing vite.config.ts — no separate Babel/Jest config needed
  - It understands import.meta.env (Vite's env variable syntax) natively
  - Module bundling and transformation are shared with the dev build, making
    it significantly faster than Jest on Vite projects

@testing-library/react handles component rendering and DOM querying. Its
philosophy is to query the DOM the way a user would — by visible text, by
role, by label — rather than by internal React component structure.

jsdom emulates a browser DOM inside Node.js so React components can render
and be asserted against without launching a real browser.

--------------------------------------------------------------------------------
2. NPM SCRIPTS
--------------------------------------------------------------------------------

  npm test                 vitest run   — one-shot CI-style run; exits with
                                          code 1 if any test fails
  npm run test:watch       vitest       — interactive watch mode; re-runs
                                          affected tests on every file save
  npm run test:coverage    vitest run --coverage
                                        — writes a text summary to the terminal
                                          and a full HTML report under dist/

3. VITEST CONFIGURATION  (vite.config.ts — "test" block)
--------------------------------------------------------------------------------

  environment:  jsdom
    Every test file gets a fake browser environment with window, document,
    localStorage, etc.

  globals:  true
    Makes describe / it / expect / vi available without explicit imports,
    matching Jest's ergonomics.

  setupFiles:  ['./src/test/setup.ts']
    Runs setup.ts once before each test file to establish global mocks.

  include:  src/**/*.{test,spec}.{ts,tsx}
    Picks up all test files recursively under src/.

  coverage.provider:  v8
    Uses Node's built-in V8 engine — faster than Istanbul, no
    instrumentation overhead.

  coverage.include:  src/hooks/**, src/lib/**, src/pages/**, src/components/**
  coverage.exclude:  src/test/**
    Reports coverage only for production source, not the test files themselves.

  NOTE: The top-level import changed from 'vite' to 'vitest/config'. This
  is required so that TypeScript understands the "test" config block.

--------------------------------------------------------------------------------
4. GLOBAL MOCK LAYER  (src/test/setup.ts)
--------------------------------------------------------------------------------

This file runs before every test file and creates a fully controlled,
hermetic environment. Five areas are mocked:

  A. Firebase SDK  (src/lib/firebase)
     ----------------------------------------
     Replaced with a plain object: { db: {}, isConfigured: false }

     Setting isConfigured: false is the key architectural lever. Every domain
     hook (useSREData, useSecurityData, etc.) checks this flag and short-
     circuits to its built-in mock fallback data immediately, without ever
     calling Firestore. This makes all hook tests synchronous — no async,
     no waitFor, no fake timers required.

  B. Firestore API  (firebase/firestore)
     ----------------------------------------
     Each Firestore function (doc, collection, onSnapshot, query, orderBy,
     setDoc, serverTimestamp) is replaced with vi.fn() stubs so no real
     network I/O fires even if a hook escapes the isConfigured guard.

     onSnapshot specifically returns a function (vi.fn()) to match
     Firestore's unsubscribe pattern — hooks that call it would otherwise
     throw on unmount.

  C. Datadog  (src/lib/datadog)  — PARTIAL mock via importActual
     ----------------------------------------
     vi.mock('../lib/datadog', async (importActual) => {
       const actual = await importActual()
       return {
         ...actual,                          // real helper functions kept
         isDatadogConfigured: false,         // overridden
         queryMetrics: vi.fn()               // overridden (makes HTTP calls)
       }
     })

     importActual spreads the real module in first, then only overrides what
     needs to be controlled. queryMetrics is stubbed because it would make
     real HTTP requests. All other helpers — latestValue, firstSeriesPoints,
     hoursAgo, daysAgo — run their actual implementations. This is why
     utils.test.ts can unit-test them as pure functions.

  D. ResizeObserver  (browser API not in jsdom)
     ----------------------------------------
     Recharts requires this API to measure chart container dimensions. jsdom
     does not include it, so a no-op class is injected globally:

       global.ResizeObserver = class ResizeObserver {
         observe()    {}
         unobserve()  {}
         disconnect() {}
       }

     Without this, every component test that renders a chart would throw.

  E. window.matchMedia  (browser API not in jsdom)
     ----------------------------------------
     useDarkMode calls window.matchMedia('(prefers-color-scheme: dark)') on
     mount. jsdom does not implement this API, so it is stubbed to return
     matches: false (light mode) with all required listener methods as no-ops.

  F. scrollIntoView  (browser API not in jsdom)
     ----------------------------------------
     SecurityDashboard's "active investigations" badge calls
     ref.current.scrollIntoView() when clicked. Stubbed on the HTMLElement
     prototype so tests can also assert it was invoked:

       window.HTMLElement.prototype.scrollIntoView = vi.fn()

6. DOM QUERY STRATEGY
--------------------------------------------------------------------------------

A deliberate hierarchy of queries avoids brittle tests:

  Unique text visible to user
    getByText('exact string')
    Fast, readable, fails if the text is absent or duplicated.

  Text appearing in multiple nodes
    getAllByText(/regex/).length > 0
    Avoids the "Found multiple elements" error that getByText throws.

  Interactive element
    getByRole('button', { name: /label/i })
    Matches the accessibility tree (computed accessible name), not DOM
    structure. Resilient to CSS and markup changes.

  Sidebar buttons (label + shortcut digit in accessible name)
    getByRole('button', { name: /engineering/i })
    Uses a partial/regex match because the button's computed accessible name
    is "Engineering 2" (label + shortcut span concatenated by the browser).

  Element that should be absent
    queryByText(...).not.toBeInTheDocument()
    getBy throws on absence; queryBy returns null, which toBeInTheDocument
    can then assert against.

--------------------------------------------------------------------------------
8. KEY DESIGN DECISIONS
--------------------------------------------------------------------------------

  No snapshots
    Snapshot tests couple assertions to rendered markup. Behaviour assertions
    (text visible, button calls handler) survive UI refactors and are more
    meaningful as regression guards.

  importActual for Datadog
    Only the side-effectful functions (network calls) are mocked. Pure
    utility functions keep their real implementations so utils.test.ts
    exercises the actual logic rather than a stub.

  isConfigured: false drives the mock data path
    The same code branch that runs in production when Firebase is unconfigured
    runs in tests. This means the tests exercise the real fallback logic and
    mock data definitions, catching regressions in both.

  ThemeContext wrapper per component test
    All page components consume useTheme(). Rather than mocking the hook,
    a real ThemeContext.Provider with a no-op toggle is used. This keeps
    the component tree valid and avoids mocking internal React hooks.

  asserting shape, not values, in hook tests
    Tests check that KPI strings end with '%' or 's', that arrays have the
    right length, and that objects contain required keys — not that MAU is
    exactly "2.4M". Specific values can change; structure should not.
